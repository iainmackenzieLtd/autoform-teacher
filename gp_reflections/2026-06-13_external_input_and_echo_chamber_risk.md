# External Input and Echo-Chamber Risk

Date: 2026-06-13  
Source: GP / ChatGPT  
Project area: AutoForm / AI Trust / Organisational Brain  
Status: Draft field note  

## Context

During the AutoForm project, Iain has been working with Claude Code as implementation partner and GP as a critical governance / strategic partner.

This creates a useful working triad, but it also carries a risk: agreement between two AI systems can feel like stronger validation than it really is.

## Core observation

Agreement between Claude Code and GP is useful signal, but not proof.

The project will eventually need external input from humans with relevant expertise and lived practical perspective.

The question should not simply be:

> Who can give feedback?

A better question is:

> What kind of reality check is this project currently missing?

## Possible external perspectives

| Person / role | What they test |
|---|---|
| Experienced developer | Architecture, security assumptions, deployment risk |
| Data / privacy person | Data handling, disclosure, retention, consent |
| HR / recruitment person | Whether workflow matches real hiring systems |
| Teacher applicant | Whether the tool genuinely reduces friction |
| School / MAT leader | Institutional risk, safeguarding, recruitment concerns |
| Governance / ethics person | Whether boundaries and controls are credible |
| Critical friend / non-technical user | Whether explanation and user experience are understandable |

## Timing

External input is important, but should not be rushed.

The project should first have:

- stable synthetic test forms
- submit blocking implemented in code
- allowlist / test mode working
- data and logging behaviour checked
- basic governance documentation
- clear explanation of what the app is and is not

## Risk / reward

Risk:
- too much external input too early may create noise, anxiety, or premature positioning

Reward:
- external input reduces echo-chamber risk and tests assumptions against reality

## Working principle

External review should be staged.

First build a small, safe, demonstrable prototype.  
Then invite targeted review from people who test specific missing perspectives.
